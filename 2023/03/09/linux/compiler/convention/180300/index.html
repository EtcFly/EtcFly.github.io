常用编译器调用约定

本文最后更新于：2024年3月5日 凌晨

版权归作者所有, 转载请保留该部分声明。
本文作者：EtcFly
原文地址：https://etcfly.top

简介

  在计算机出现之前，机器制造商通常会为多种编程语言提供操作系统和编译器, 每个平台的调用约定是由制造商的编程工具定义的。随着技术越来越复杂, 技术分工越来越明细, 操作系统和编译器维护的难度越来越越大, 届时许多独立的软件公司提供操作系统、多种编程语言编译器和应用程序，设备厂商负责机器生产的社会格局开始形成。

  但是不同公司的调用方案各不相同，各种编译器实现这些约定的方式存在细微差别，因此通常很难连接由不同编译器编译的代码,这些方案相互排斥，很难相互兼容。最终经过市场洗牌之后，Microsoft操作系统和编程工具（具有不同的约定）占据主导地位，而Borland和Novell等二线公司以及GCC等开源项目仍然保持着自己的标准。供应商和产品之间的互操作性条款最终被采纳，简化了选择可行约定的问题。

调用标准主要约定了如下几部分内容:

  • 函数内分配原子变量和复杂变量各部分的顺序。
  • 函数调用参数如何传输(参数是压入堆栈还是放入寄存器, 或者二者都存在)
  • 被调用函数需要为调用函数保留那些寄存器
  • 函数调用前后的上下文恢复(在调用者和被调用者中如何保存和恢复现场)

调用约定

调用者清理现场(Caller clean-up)

  调用者从堆栈中清除参数, 主要是局部变量和函数上下文。这类调用主要分为如下几种:

  • cdecl
  • syscall
  • optlink
cdecl

  cdecl通常是x86 c编译器的默认调用约定, 许多C编译器都将其用于x86体系结构。许多编译器还提供了自动更改所使用的调用约定的选项：

其主要特征如下:

  • 子程序参数在堆栈上传递
  • 子函数参数压栈从右到左的顺序进行
  • 调用者清理被调用者的堆栈现场(子程序返回恢复老的调用者现场)
  • EAX寄存器用于返回整数值或者内存地址
  • 浮点数据使用ST0 x87寄存器返回
  • EAX ECX EDX寄存器由调用者保存, 其他寄存器由被调用者保存

cdecl后期又有了一些更新, 主要是为了解决不同操作系统平台或由不同编译器编译的 x86 程序可能的不兼容:

  • 返回者(为了大数据结构或者类对象在内存中返回, 调用者分配内存地址并将其指向它的指针作为隐藏的第一个参数传递，被调用者填充内存并返回指针)
syscall

  和cdecl约定类似, 其特征如下:

  • 参数从右到左传递压栈
  • 调用者不保存EAX ECX EDX寄存器, 双字参数列表的大小在AL寄存器传递

syscall是OS/2系统的标准调用约定。

optlink

主要特征如下:

  • 参数从右到左传递压栈
  • 函数前三个参数(从左边开始)在EAX ECX EDX中传递
  • 最多四个浮点参数在 ST0 到 ST3 中传递
  • 调用结果在 EAX 或 ST0 中返回

Optlink 由 IBM VisualAge 编译器使用

被调用者清理现场(Callee clean-up)

  被调用者从堆栈中清除参数,并返回给调用者。其主要类型有:

  • pascal
  • stdcall
  • fastcall
  • vectorcall
pascal

  基于Borland Pascal编程语言的调用约定, 主要特征如下:

  • 参数按从左到右的顺序压入堆栈（与 cdecl 相反）
  • 被调用者负责局部变量从堆栈中移除
  • 返回值约定
    • 序数值以 AL（8位值）、AX（16 位值）、EAX（32 位值）或 DX:AX（16 位系统上的 32 位值）返回
    • 实际值在 DX:BX:AX 中返回。
    • 在 ST0 中返回浮点 (8087) 值
    • 指针在 32 位系统的 EAX 和 16 位系统的 AX 中返回
    • 字符串在@Result 符号指向的临时位置返回

这种调用约定在 16 位 API 中很常见：OS/2 1.x、Microsoft Windows 3.x 和 Borland Delphi 1.x 版。Windows API 的现代版本使用stdcall，它仍然让被调用方按照 Pascal 约定恢复堆栈，但参数现在从右向左推送。

stdcall

  stdcall调用约定是 Pascal调用约定的变体，区别在于:

  • 参数按从右到左的顺序压入堆栈，如 _cdecl 调用约定。
  • 寄存器 EAX、ECX 和 EDX 指定用于函数内。
  • 返回值存储在 EAX 寄存器中

stdcall 是 Microsoft Win32 API 和 Open Watcom C++ 的标准调用约定

fastcall

  Microsoft __fastcall约定（又名__msfastcall）, 主要特征:

  • 将适合的前两个参数（从左到右计算）传递到 ECX 和 EDX, 剩余的参数从右到左压入堆栈。

当编译器针对IA64或AMD64进行编译时，它会忽略__fastcall关键字并改为使用一个 64 位调用约定, 其他编译器，如GCC、Clang和ICC 提供类似的 fastcall 调用约定，尽管它们不一定相互兼容或与 Microsoft fastcall 兼容。

vectorcall

  在Visual Studio 2013 中，Microsoft引入了__vectorcall调用约定，以响应游戏、图形、视频/音频和编解码器开发人员的效率问题, 该方案允许较大的向量类型（float、double、__m128、__m256）在寄存器中传递，而不是在堆栈中传递。

调用者或被调用者清理现场(caller or callee clean-up)

主要是针对thiscall

thiscall

  此调用约定用于调用 C++ 非静态成员函数。thiscall有两个主要版本，具体取决于编译器以及函数是否使用可变数量的参数。

  • GCC编译器: thiscall几乎与cdecl相同
    • 调用者清理堆栈
    • 参数按从右到左的顺序传递
    • 不同之处在于添加了this指针，它最后被压入堆栈，就好像它是函数原型中的第一个参数一样。
  • Microsoft Visual C++ 编译器: 类似于stdcall
    • this指针在 ECX 中传递时, 被调用者清理堆栈， 类似与stdcall
    • 当函数使用可变数量的参数时，清理堆栈的是调用者, 类似于cdecl

总结

  调用约定是为了应对技术的发展以及日益变化的需求而约定的规则，原则上使用汇编语言开发不存在任何调用规范的问题。调用规范本质上有几个目的:

  • 提供程序调用的效率(提高调用引起的效率降低)
  • 为高级语言翻译成汇编语言提供规范标准
  • 为各平台的编译程序提供兼容(如不同编译器编译的库文件)，有利于系统和编译器等各部分软件的分工。
