http协议之优化技术

本文最后更新于：2024年3月5日 凌晨

版权归作者所有, 转载请保留该部分声明。

简介

  由于http协议基于客户端-服务器模型, 每一个连接必须由客户端发起, 且一次连接完成服务端不保留连接状态, 这就导致很多性能的问题。如下是http存在的一些常见问题:

  • 新tcp连接存在慢启动
  • 新tcp链接握手时延较大
  • 服务端进行并发处理压力较大
  • 连接无状态, 导致前后请求无法关联
  • http传输的资源很难进行管控

基于以上种种的缺点, http协议引入了token, cookie, session, 持久连接、缓存、代理等等机制。现在来逐个了解。

HTTP优化技术

  由于日益复杂的用户需求, 网页变得越来越复杂, 传统的http无法满足人们的需求, 于是重新设计了http2.0协议, 但是http1.1协议仍然是使用最广泛的http协议版本， 为了兼容老的应用, 协议者们纷纷在主流协议版本上添加了一些策略来应对新的需求, 提高体验。策略包括:

  • 引入缓存(session, cookie)
  • 引入代理(实现内容过滤, 并发访问等, 如现在主流CDN就是进行资源缓存)
  • 连接复用(http结束不在断开连接, 而是使用长连接)
  • 在传统http上协商websocket协议

我们来逐个讲解、

连接复用

  连接管理是一个 HTTP 的关键话题：打开和保持连接在很大程度上影响着网站和 Web 应用程序的性能。在 HTTP/1.x 里有多种模型：

  • 短连接
  • 长连接
  • HTTP 流水线。

  HTTP 的传输协议主要依赖于 TCP 来提供从客户端到服务器端之间的连接。在早期，HTTP 使用一个简单的模型来处理这样的连接。这些连接的生命周期是短暂的：每发起一个请求时都会创建一个新的连接，并在收到应答时立即关闭。

  这个简单的模型对性能有先天的限制：打开每一个 TCP 连接都是相当耗费资源的操作。客户端和服务器端之间需要交换好些个消息。当请求发起时，网络延迟和带宽都会对性能造成影响。现代浏览器往往要发起很多次请求（十几个或者更多）才能拿到所需的完整信息，证明了这个早期模型的效率低下。

  有两个新的模型在 HTTP/1.1 诞生了。首先是长连接模型，它会保持连接去完成多次连续的请求，减少了不断重新打开连接的时间。然后是 HTTP 流水线模型，它还要更先进一些，多个连续的请求甚至都不用等待立即返回就可以被发送，这样就减少了耗费在网络延迟上的时间。

  要注意的一个重点是 HTTP 的连接管理适用于两个连续节点之间的连接，它是逐跳（Hop-by-hop）标头，而不是端到端（End-to-end）标头。当模型用于从客户端到第一个代理服务器的连接和从代理服务器到目标服务器之间的连接时（或者任意中间代理）效果可能是不一样的。HTTP 协议头受不同连接模型的影响，比如 Connection 和 Keep-Alive，就是逐跳标头标头，它们的值是可以被中间节点修改的。

短连接

  HTTP 最早期的模型和 HTTP/1.0 的默认模型，是短连接。每一个 HTTP 请求都由它自己独立的连接完成；这意味着发起每一个 HTTP 请求之前都会有一次 TCP 握手，而且是连续不断的。

  TCP 协议握手本身就是耗费时间的，所以 TCP 可以保持更多的热连接来适应负载。短连接破坏了 TCP 具备的能力，并且新的冷连接降低了其性能。

  这是 HTTP/1.0 的默认模型（如果没有指定 Connection 协议头，或者是值被设置为 close）。而在 HTTP/1.1 中，只有当 Connection 被设置为 close 时才会用到这个模型。

长连接

  短连接有两个比较大的问题：创建新连接耗费的时间尤为明显，另外 TCP 连接的性能只有在该连接被使用一段时间后（热连接）才能得到改善。为了缓解这些问题，长连接的概念便被设计出来了，甚至在 HTTP/1.1 之前。或者，这被称之为一个 keep-alive 连接。

  一个长连接会保持一段时间，重复用于发送一系列请求，节省了新建 TCP 连接握手的时间，还可以利用 TCP 的性能增强能力。当然这个连接也不会一直保留着：连接在空闲一段时间后会被关闭（服务器可以使用 Keep-Alive 协议头来指定一个最小的连接保持时间）。

  长连接也还是有缺点的；就算是在空闲状态，它还是会消耗服务器资源，而且在重负载时，还有可能遭受 DoS 攻击。这种场景下，可以使用非长连接，即尽快关闭那些空闲的连接，也能对性能有所提升。

  HTTP/1.0 里默认并不使用长连接。把 Connection 设置成 close 以外的其他参数都可以让其保持长连接，通常会设置为 retry-after。在 HTTP/1.1 里，默认就是长连接的，不再需要标头（但我们还是会把它加上，万一某个时候因为某种原因要退回到 HTTP/1.0 呢）。

HTTP 流水线

  默认情况下，HTTP 请求是按顺序发出的。下一个请求只有在当前请求收到响应过后才会被发出。由于会受到网络延迟和带宽的限制，在下一个请求被发送到服务器之前，可能需要等待很长时间。

  流水线是在同一条长连接上发出连续的请求，而不用等待应答返回。这样可以避免连接延迟。理论上讲，性能还会因为两个 HTTP 请求有可能被打包到一个 TCP 消息包中而得到提升。就算 HTTP 请求不断的继续，尺寸会增加，但设置 TCP 的最大分段大小（MSS）选项，仍然足够包含一系列简单的请求。

  并不是所有类型的 HTTP 请求都能用到流水线：只有幂等方式，比如 GET、HEAD、PUT 和 DELETE 能够被安全地重试。如果有故障发生时，流水线的内容要能被轻易的重试。

  今天，所有遵循 HTTP/1.1 标准的代理和服务器都应该支持流水线，虽然实际情况中还是有很多限制：一个很重要的原因是，目前没有现代浏览器默认启用这个特性。

代理

  现代的web分层结构中, 代理是一个很重要的概念, 引入代理具有很多的好处。

  • 提供资源的访问速度

缓存服务器可以将远端服务端的资源进行缓存, 后期再次访问服务端可以直接从缓存获取数据, 而不在走因特网请求数据。访问速度会很快。

  • 进行资源权限的访问

对于一些公司内部的文档或者资源, 可以通过代理服务器进行权限审查和管理。

  • 对不良信息进行过滤

对于一些儿童类网站, 可以进行内容过滤, 避免污秽内容。

  • 隐藏设备真实ip

可以以代理服务器为跳板访问目的服务器, 起到隐藏ip访问。

  • 缓解服务器的并行资源压力

缓存远端服务器资源, 后期请求资源直接从代理缓存服务器获取数据而不直接向原始服务器请求。

如上是一个简单的代理结构, 复杂的代理可能会存在多级结构。代理还引入了很多代理的头部, 具体请参考相关代理头部内容。

缓存技术

  缓存技术是一个很大的话题, 数据缓存可以解决很多的性能问题:

  • 设备带宽瓶颈
  • 瞬时拥塞
  • 冗余的数据传输
  • 较远距离的时延

  可以毫不夸张的说, 缓存服务器是整个web服务的基石, 一般来说缓存处理数据的流程如下:

  1. 接受: 缓存服务器从网络读取抵达的请求报文
  2. 解析: 缓存对报文进行解析, 提取url和首部
  3. 查询: 缓存检查本地是否存在副本, 没有, 缓存服务器向目标请求一份副本, 并缓存起来。
  4. 新鲜度监测: 缓存查看以缓存副本是否足够新鲜, 否则重新请求一份副本
  5. 创建响应: 缓存服务器使用新首部和缓存主体构建一条响应报文
  6. 发送: 缓存通过网络将响应发给客户端
  7. 日志: 缓存服务器生成一条日志记录。

关于如何监测文档的新鲜度, 一般是利用相关头部描述的文档生命周期来检测:

  • Cache-Control: max-age: 文档缓存的秒数
  • Expires: <日期>: 文档缓存有效的最后时间点

另外缓存服务器还可以使用条件验证:

  • If-Modified-Since: <date>: 如果从指定日期之后文档被修改则执行请求方法。
  • If-None-Match: <tags>: 提供细颗粒度的文档表情, 避免文档简单的保存导致重复缓存。

具体缓存策略请参考相关文档。

token

  为了安全考虑, 客户端请求一些受限资源的时候需要提供身份认证, 以此允许服务端来鉴别客户端是否是一个合法设备来判断设备是否有权限访问和执行特定命令或者资源, 如果每次客户端访问都带上用户名和密码会导致一些风险:

  • 密码泄漏的风险
  • 增加冗余的流量
  • 减少服务端压力

  因此, 服务端为了解决这个问题, 引入了Token机制。客户端首次访问特定接口或者资源时, 携带用户名和密码等认证数据向服务端请求权限, 这时候服务端会返回一个Token, 后续客户端便可以携带Token向服务端执行请求操作。当然Token还需要设置一个过期期限, 过期后Token将在服务器丢弃失效。

  这里还存在一个问题, 客户端使用token可能会被第三方截获并进行报文重放, 因此Token还需要考虑重放和有效期等问题。一些典型做法有:

  1. 请求时携带Token和客户端引入唯一的nonce, 每次值不一样, 服务端如果收到相同nonce, 直接丢弃, 当然还需要使用客户端和服务端唯一的密钥来对提供的Token,nonce等参数做签名防止篡改。
  2. 携带Token和时间戳, 相同时间错服务端将丢弃处理。同样需要加入MAC做校验。

  其次, 对于服务端, 基于Token机制, 服务端避免存储客户端的回话状态信息, 避免了查询数据库等开销。具体步骤:

  1. 客户端使用用户名跟密码请求登录
  2. 服务端收到请求，去验证用户名与密码
  3. 验证成功后，服务端会签发一个 Token，再把这个 Token 发送给客户端
  4. 客户端收到 Token 以后可以把它存储起来，比如放在 Cookie 里或者数据库里
  5. 客户端每次向服务端请求资源的时候需要带着服务端签发的 Token
  6. 服务端收到请求，然后去验证客户端请求里面带着的 Token，如果验证成功，就向客户端返回请求的数据

session

  早期服务端通过Session来保存和客户端之间的回话, 它是服务端早期保存回话的一种数据结构(主要存储的的SessionID和Session内容，同时也包含了很多自定义的内容如：用户基础信息、权限信息、用户机构信息、固定变量等),这个数据可以保存在集群、数据库、文件中，用于跟踪用户的状态。

  客户端浏览器访问服务器的时候，服务器把客户端信息以某种形式记录在服务器上。这就是Session。客户端浏览器再次访问时只需要从该Session中查找该客户的状态就可以了。

  用户第一次登录后，浏览器会将用户信息发送给服务器，服务器会为该用户创建一个SessionId，并在响应内容（Cookie）中将该SessionId一并返回给浏览器，浏览器将这些数据保存在本地。当用户再次发送请求时，浏览器会自动的把上次请求存储的Cookie数据自动的携带给服务器。

服务器接收到请求信息后，会通过浏览器请求的数据中的SessionId判断当前是哪个用户，然后根据SessionId在Session库中获取用户的Session数据返回给浏览器。

  当然Session并不是长期有效, 一般服务端会记录客户端每次使用SessionId最后访问时间, 当长时间没访问时, 服务端会删除这些不活跃的SessionId。

cookie

  可以将Cookie分为两类: 回话持久cookie和持久cookie, 回话cookie是一种临时cookie, 它记录用户访问站点时的设置和偏好, 用户退出浏览器时会被删除。持久cookie生存时间更长, 它存储在硬盘上, 并不会随着计算机重启而丢失。

  回话cookie和持久cookie的唯一区别就是过期时间, 如果没有说明过期时间则说明是一个回话cookie。

一般cookie存储在浏览器的cookie数据库, 包含如下常用项:

  • domain: cookie的域

Domain 指定了哪些主机可以接受 Cookie。如果不指定，该属性默认为同一 host 设置 cookie，不包含子域名。如果指定了 Domain，则一般包含子域名。因此，指定 Domain 比省略它的限制要少。但是，当子域需要共享有关用户的信息时，这可能会有所帮助。

  • path: 相关的路径

Path 属性指定了一个 URL 路径，该 URL 路径必须存在于请求的 URL 中，以便发送 Cookie 标头。以字符 %x2F (“/”) 作为路径分隔符，并且子路径也会被匹配。

  • name-value: cookie对应的name和value
  • expiration: 过期时间

安全和认证

  由于http协议是明文的, 很容易被第三方截获和篡改, 因此对于一些私密性的回话是不安全, 早期http通过将内容进行base64来转码, 但是同样不够安全。后来又引入了摘要计算, 但是同样不够安全。现代应用已经广泛并使用tls来构建安全的传输环境。

  tls通过在tcp层之上构建一些加密层, 对应用层相关协议数据进行高强度的加密来达到安全目的, 目前主流的是tls1.2, 但是最新的tls1.3已经废弃了tls1.2, 因此推荐使用最新的tls1.3协议，关于tls协议请参考 tls1.3协议原理
