mqtt5协议一简介

1. 背景

  自从mqtt v3.1.1标准化以后, 各物联网厂商迅速对其进行了支持, 使得mqtt v3.1.1协议被广泛应用在各行各业, 随之而来也暴露了很多问题, 比如之前协议对会话保留约定粗糙的颗粒导致服务端大量的资源消耗, 为了解决mqtt v3.1.1协议的不足, OASIS 组织重新推出了mqtt v5.0协议, 有效解决了现存的问题。

  协议层面， v5.0保持了对v3.1.1协议的完全兼容, 也就是说v5.0协议是v3.1.1的超集, 其扩展部分主要包括: 新增了属性, AUTH包, 共享订阅等一系列机制。OASIS 组织针对v3.1.1协议的资源消耗、扩展性等方面做了全面改进, 在新版本的 v5.0协议中主要做了如下改变:

  • 增强了大规模系统可伸缩性(共享订阅等机制)
  • 为服务中异常情况提供了丰富的错误报告
  • 规范化了容量探索和请求响应在内的通用模式(最大包长度、最大接受包等)
  • 包括用户属性在内的可扩展机制(丰富的属性字段)
  • 改进性能支持小型客户端(主题别名、订阅标识符、会话过期间距等机制)
  • 增强的安全认证机制(新增AUTH交互报文)

接下来就让我们一起来学习mqtt v5.0 协议吧！

  • RETAIN: PUBLISH 报文的保留标志
2.1.3 剩余长度

  剩余长度字段使用一个变长度编码方案， 对小于 128 的值它使用单字节编码。 更大的值按下面的方式处理。低 7 位有效位用于编码数据，最高有效位用于指示是否有更多的字节。 因此每个字节可以编码 128 个数值和一个延续位（continuation bit） 。 剩余长度字段最大 4 个字节。

变长编码算法:

MOD 是模运算， DIV 是整数除法， OR 是位操作或（C 语言中分别是%， /， |）

变长解码算法:

AND 是位操作与（C 语言中的 &）, 这个算法终止时， value 包含的就是剩余长度的值。

2.2 可变报头

  可变报头一般由 属性 和 报文标识符 组成, 但是并不是所有报文都包含。MQTT 控制报文的可变报头部分包含了 2 个字节的报文标识符字段。这些 MQTT 控制报文类型为：PUBLISH 报文（当 QoS>0 时）， PUBACK， PUBREC， PUBREC， PUBREL， PUBCOMP，SUBSCRIBE， SUBACK， UNSUBSCRIBE， UNSUBACK。

2.1.2 属性

  属性字段由 属性长度 和 其后的 属性 组成。属性长度被编码为变长字节整数, 属性长度不包含编码属性长度自身字节数, 属性长度仅仅包含跟随其后的所有属性的长度总和。

  一个属性包含一段数据和一个定义了属性用途和数据类型的标识符。 标识符被编码为变长字节整数。任何控制报文，如果包含了：对于该报文类型无效的标识符，或者错误类型的数据，都是无效报文。收到无效报文时，服务端或客户端使用包含原因码0x81（无效报文） CONNACK 或 DISCONNECT 报文进行错误处理。

2.3 有效载荷

  某些 MQTT 控制报文在报文的最后部分包含一个有效载荷, 对于 PUBLISH 来说有效载荷就是应用消息。

2.4 原因码

  原因码是一个单字节无符号数，用来指示一次操作的结果。小于 0x80 的原因码指示某次操作成功完成，通常用 0 来表示。大于等于 0x80的原因码用来指示操作失败。CONNACK， PUBACK， PUBREC， PUBREL， PUBCOMP， DISCONNECT 和 AUTH 控制报文的可变报头有一个单字节的原因码。 SUBACK 和 UNSUBACK 报文的载荷字段包含一个或多个原因码。

  • 对于原因码 0x91（报文标识符已被占用）的处理可以为尝试修复会话、以新会话标志为 1 重置会话或者判定客户端或服务端实现有缺陷。
